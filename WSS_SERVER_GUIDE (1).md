# WSS Server Integration Guide

This document explains how to build the **server side** that communicates with ModemGrid PRO instances via WebSocket Secure (WSS).

## Overview

```
┌──────────────────┐       WSS        ┌──────────────────┐
│   YOUR SERVER    │ ◄──────────────► │  MODEMGRID NODE  │
│  (orchestrator)  │                  │  (dongle runner)  │
│                  │   JSON messages  │                   │
│  - Receives API  │ ──────────────►  │  - Executes USSD  │
│    requests      │                  │  - Manages dongles│
│  - Sends topup   │  ◄────────────── │  - Returns result │
│    commands      │   result + UUID  │                   │
└──────────────────┘                  └──────────────────┘
```

The server is the **orchestrator**. Multiple ModemGrid nodes can connect to it. Each node has dongles attached and waits for the server to send topup requests.

---

## Connection Lifecycle

### 1. Client Connects

The ModemGrid node opens a WSS connection to your server:

```
wss://your-server.com/ws/client
Authorization: Bearer <token>
```

### 2. Registration

Immediately after connecting, the node sends a registration message with its full status (pools, dongles, balances):

```json
{
  "type": "register",
  "token": "<auth_token>",
  "pools": [...],
  "dongles": [...],
  "dongle_count": 3,
  "online_count": 2
}
```

See [Node Status Reporting](#node-status-reporting) below for the full `pools` and `dongles` structure.

The server should:
- Validate the token
- Track this connection as an active node
- Store the node's pool/dongle status for routing decisions
- Optionally send a welcome message: `{"type": "welcome"}`

### 3. Heartbeat (optional)

The node's WebSocket library sends ping frames every 30s. The server should respond with pong (handled automatically by most WS libraries).

### 4. Disconnect / Reconnect

If the connection drops, the node auto-reconnects after `reconnect_interval` seconds (default 5s, configurable per node). The server should:
- Mark the node as offline
- Clean up any pending requests
- Accept the reconnection when it comes back

## Message Protocol

### Server → Node: Topup Request

The server sends a topup command to the node:

```json
{
  "api_name": "test_var",
  "variables": {
    "phone_number": "77000",
    "price": "10"
  }
}
```

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `api_name` | string | Yes | Name of the API definition configured on the node (e.g. `"test_var"`, `"sold_mobilis"`) |
| `variables` | object | No | Key-value pairs matching the API's defined variables |

**No `request_id` is sent.** The node generates its own UUID and returns it in the response.

### Node → Server: Result

After executing the topup (or on failure), the node sends back:

#### Success
```json
{
  "request_id": "4b3ea361-8075-4e8a-b988-5ebf1a768a09",
  "status": "completed",
  "result": "Topup successful. New balance: 15.00 DA",
  "error": null,
  "modem_id": "dongle3"
}
```

#### Failed
```json
{
  "request_id": "4b3ea361-8075-4e8a-b988-5ebf1a768a09",
  "status": "failed",
  "result": null,
  "error": "Insufficient balance: 5.0 < topup amount 10",
  "modem_id": "dongle3"
}
```

#### Timeout
```json
{
  "request_id": "4b3ea361-8075-4e8a-b988-5ebf1a768a09",
  "status": "timeout",
  "result": null,
  "error": "Pool execution timed out (180s)",
  "modem_id": null
}
```

| Field | Type | Description |
|-------|------|-------------|
| `request_id` | string (UUID) | Generated by the node. Use this to match responses to requests. |
| `status` | string | `"completed"`, `"failed"`, or `"timeout"` |
| `result` | string \| null | Final USSD response text on success |
| `error` | string \| null | Error message on failure |
| `modem_id` | string \| null | Which dongle handled the request (e.g. `"dongle3"`) |

### Node → Server: Status Update (periodic)

Every 30 seconds, the node sends a status update with current pool/dongle data:

```json
{
  "type": "status",
  "pools": [...],
  "dongles": [...],
  "dongle_count": 3,
  "online_count": 2
}
```

The server should use this to keep its view of each node's capacity up to date.

## Node Status Reporting

The `pools` and `dongles` arrays in both `register` and `status` messages contain detailed information about the node's dongles and their balances.

### `dongles` Array

Each entry represents one dongle:

```json
{
  "dongle_id": "dongle3",
  "name": "MOBILIS dongle3",
  "operator": "ALG Mobilis",
  "online": true,
  "balance": 25.0,
  "pool_ids": [1, 2]
}
```

| Field | Type | Description |
|-------|------|-------------|
| `dongle_id` | string | Unique dongle identifier (e.g. `"dongle3"`) |
| `name` | string | Human-readable name |
| `operator` | string | Operator name (e.g. `"ALG Mobilis"`, `"Djezzy"`) |
| `online` | boolean | Whether the dongle is currently connected and responsive |
| `balance` | float \| null | Current balance parsed from USSD credit check (null if unknown) |
| `pool_ids` | int[] | IDs of pools this dongle belongs to |

### `pools` Array

Each entry represents one dongle pool:

```json
{
  "pool_id": 1,
  "name": "Mobilis Pool",
  "dongle_count": 3,
  "online_count": 2,
  "total_balance": 65.0,
  "highest_balance": 25.0,
  "dongles": [
    {
      "dongle_id": "dongle3",
      "name": "MOBILIS dongle3",
      "operator": "ALG Mobilis",
      "online": true,
      "balance": 25.0
    },
    {
      "dongle_id": "dongle4",
      "name": "MOBILIS dongle4",
      "operator": "ALG Mobilis",
      "online": true,
      "balance": 20.0
    },
    {
      "dongle_id": "dongle5",
      "name": "MOBILIS dongle5",
      "operator": "ALG Mobilis",
      "online": false,
      "balance": 20.0
    }
  ],
  "apis": ["topup_mobilis", "sold_mobilis"]
}
```

| Field | Type | Description |
|-------|------|-------------|
| `pool_id` | int | Unique pool ID |
| `name` | string | Pool name |
| `dongle_count` | int | Total dongles in pool |
| `online_count` | int | How many dongles are currently online |
| `total_balance` | float | Sum of all dongle balances in pool |
| `highest_balance` | float | Highest single dongle balance in pool |
| `dongles` | array | Per-dongle details (same structure as `dongles` array above, minus `pool_ids`) |
| `apis` | string[] | API names assigned to this pool |

### Top-level Status Fields

| Field | Type | Description |
|-------|------|-------------|
| `dongle_count` | int | Total active dongles on this node |
| `online_count` | int | How many dongles are online |

### How the Server Can Use This Data

- **Balance-based routing**: Route topup requests to the node/pool with the highest `total_balance` or `highest_balance`
- **Capacity checking**: Check `online_count` > 0 before sending a request to a pool
- **API availability**: Use the `apis` array to know which `api_name` values a pool can handle
- **Operator routing**: Use `operator` field to route to the right pool (e.g. Mobilis topups go to Mobilis pools)
- **Dongle health**: Track `online` status to avoid routing to offline dongles

### Server → Node: Non-Topup Messages (ignored)

The node ignores these message types from the server:
- `{"type": "heartbeat"}`
- `{"type": "ack"}`
- `{"type": "welcome"}`

## Server-Side Implementation (Python Example)

Using `websockets` library:

```python

import websockets
from websockets.server import serve

# Track connected nodes and their status

async def handle_client(websocket):
    # Register this node
    connected_nodes.add(websocket)
    print(f"Node connected. Total: {len(connected_nodes)}")

    try:
        async for message in websocket:
            data = json.loads(message)

            # Handle registration (includes pools + dongles status)
            if data.get("type") == "register":
                token = data.get("token")
                # Validate token here
                pools = data.get("pools", [])
                dongles = data.get("dongles", [])
                print(f"Node registered: {len(dongles)} dongles, {len(pools)} pools")
                # Store node status for routing decisions
                node_status[websocket] = {"pools": pools, "dongles": dongles}
                await websocket.send(json.dumps({"type": "welcome"}))
                continue

            # Handle periodic status update
            if data.get("type") == "status":
                pools = data.get("pools", [])
                dongles = data.get("dongles", [])
                print(f"Status update: {len(dongles)} dongles, {len(pools)} pools")
                # Update stored node status
                node_status[websocket] = {"pools": pools, "dongles": dongles}
                continue

            # Handle result from a previous topup request
            if "request_id" in data:
                print(f"Result received: {data}")
                # Match request_id to your original request
                # Store result in your database
                continue

    except websockets.exceptions.ConnectionClosed:
        print("Node disconnected")
    finally:
        connected_nodes.discard(websocket)

    """Send a topup request to a specific node."""

        "api_name": api_name,
        "variables": variables,

    await websocket.send(json.dumps(request))

    # Handle it in the handle_client loop above


async def main():
    async with serve(handle_client, "0.0.0.0", 8760):
        print("WSS server listening on :8760")
        await asyncio.Future()  # run forever

asyncio.run(main())
```

## Server-Side Implementation (Node.js Example)

Using `ws` library:

```javascript
const { WebSocketServer } = require('ws');

const wss = new WebSocketServer({ port: 8760 });
const connectedNodes = new Set();
const nodeStatus = new Map();  // ws -> { pools: [...], dongles: [...] }

wss.on('connection', (ws) => {
    connectedNodes.add(ws);
    console.log(`Node connected. Total: ${connectedNodes.size}`);

    ws.on('message', (data) => {
        const msg = JSON.parse(data);

        // Handle registration (includes pools + dongles status)
        if (msg.type === 'register') {
            console.log(`Node registered: ${msg.dongle_count} dongles, ${msg.pools?.length} pools`);
            // Store node status for routing
            nodeStatus.set(ws, { pools: msg.pools, dongles: msg.dongles });
            ws.send(JSON.stringify({ type: 'welcome' }));
            return;
        }

        // Handle periodic status update
        if (msg.type === 'status') {
            console.log(`Status update: ${msg.dongle_count} dongles, ${msg.pools?.length} pools`);
            nodeStatus.set(ws, { pools: msg.pools, dongles: msg.dongles });
            return;
        }

        // Handle result
        if (msg.request_id) {
            console.log(`Result: ${msg.status} - ${msg.result || msg.error}`);
            // Match request_id and store result
            return;
        }
    });

    ws.on('close', () => {
        connectedNodes.delete(ws);
        console.log('Node disconnected');
    });
});

// Send a topup to a node
function sendTopup(ws, apiName, variables) {
    ws.send(JSON.stringify({ api_name: apiName, variables }));
}

console.log('WSS server listening on :8760');
```

## Multi-Node Setup

You can have multiple ModemGrid nodes connected to the same server:

```
                    ┌──────────────┐
                    │  YOUR SERVER │
                    └──────┬───────┘
                           │
              ┌────────────┼────────────┐
              │            │            │
     ┌────────┴───┐ ┌─────┴────┐ ┌─────┴────┐
     │  Node A    │ │  Node B  │ │  Node C  │
     │ (3 dongles)│ │ (2 dong.)│ │ (5 dong.)│
     │  Mobilis   │ │  Djezzy  │ │  Ooredoo │
     └────────────┘ └──────────┘ └──────────┘
```

Each node has its own:
- Dongles (with balances)
- API definitions
- Pools
- WSS config (URL, token, enabled)

The server routes requests to the appropriate node based on:
- The `api_name` (each node has its own API definitions)
- Or custom routing logic (by operator, region, etc.)

## Security

### Authentication
- Each node sends a Bearer token in the HTTP header on connect
- The same token is sent in the `register` message
- The server should validate this token before accepting requests

### Transport Security
- Use `wss://` (not `ws://`) for encrypted connections
- Use a valid SSL certificate (Let's Encrypt, etc.)
- The node verifies the certificate by default

### Recommended
- Use unique tokens per node
- Rotate tokens periodically
- Restrict by IP if possible
- Log all requests and results

## API Names

The `api_name` in the request must match an API definition configured on the node. Common examples:

| api_name | Description | Variables |
|----------|-------------|-----------|
| `test_var` | Test topup | `phone_number`, `price` |
| `sold_mobilis` | Check balance | (none) |
| `topup_mobilis` | Mobilis topup | `phone`, `amount`, `pin` |
| `topup_djezzy` | Djezzy topup | `phone`, `amount` |
| `topup_ooredoo` | Ooredoo topup | `phone`, `amount` |

The server must know which API names are available on each node. This can be configured manually or discovered via a future API endpoint.

---

## Error Handling

The server should handle these scenarios:

| Scenario | What happens | Server action |
|----------|-------------|---------------|
| Node offline | Connection refused / dropped | Queue request, retry when node reconnects |
| Node busy | All dongles in use | Node returns `status: "failed"` with "All dongles busy" |
| Insufficient balance | No dongle has enough credit | Node returns `status: "failed"` with "No balance in pool" |
| USSD timeout | Dongle didn't respond | Node returns `status: "timeout"` |
| API not found | `api_name` doesn't exist on node | Node returns `status: "failed"` with "API not found" |
| Execution error | Unexpected exception | Node returns `status: "failed"` with error message |

## Flow Summary

```
1. Node connects to server via WSS
2. Node sends {"type": "register", "token": "...", "pools": [...], "dongles": [...]}
3. Server validates, stores node status, sends {"type": "welcome"}
4. Every 30s, node sends {"type": "status", "pools": [...], "dongles": [...]} (same structure)
5. Server sends topup request:
   {"api_name": "test_var", "variables": {"phone_number": "77000", "price": "10"}}
6. Node executes:
   - Finds API definition
   - Routes through pool or single dongle
   - Checks balance (stored, rechecks via USSD if needed)
   - Executes USSD steps
   - Deducts balance
7. Node sends result:
   {"request_id": "uuid", "status": "completed", "result": "...", "modem_id": "dongle3"}
8. Server matches request_id, stores result
9. Repeat from step 5 (and step 4 for status updates)
```
